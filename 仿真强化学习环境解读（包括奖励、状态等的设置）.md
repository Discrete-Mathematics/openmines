# 基于强化学习与离散事件仿真的露天矿卡运输调度优化

## 摘要

在现代露天矿山开采作业中，矿用卡车的运输成本通常占总运营成本的一半以上。因此，如何高效调度卡车在各个装载区（电铲）与卸载区（排土场、破碎站）之间运转，最大化总产量并最小化排队与空耗时间，是矿山智能化升级中的难题。传统的基于规则（Rule-based）或启发式（Heuristic）的调度算法往往难以应对矿山环境中的动态不确定性（如道路拥堵、车辆故障、天气变化等）。

该项目采用了**强化学习（Reinforcement Learning, RL）**与**离散事件仿真（Discrete Event Simulation, DES）**相结合的方法。每当一辆卡车到达某个决策点（如刚出充电站、刚装完矿、刚卸完矿），系统会立刻呼叫调度器，决定其下一个目的地。这个局部的决策将引发连锁反应，直接影响后续几十分钟乃至全局的运行效率：包括排队时间、道路拥堵概率、铲车空闲时间、卸载机等待时间以及最终的全局总产量。

### 4. 车辆故障采样

基于真实设备的可靠性理论，卡车的故障时间间隔遵循指数分布，而维修时间则遵循正态分布。

```python
    def sample_breakdown(self):
        """
        从指数分布中采样故障时间。
        用于判断卡车在当前时刻是否发生突发性车辆故障，并计算所需的维修时间。
        :return: None
        """
        # 使用指数分布计算故障发生的时间（载具正常工作时长）
        time_to_breakdown = np.random.exponential(self.expected_working_time_without_breakdown)
        
        # 判断载具是否发生故障（当前时间超过故障时间）
        if self.env.now >= self.last_breakdown_time + time_to_breakdown:
            # 发生故障，使用正态分布N(mu=10, sigma=3)采样获得维修时间
            repair_time = np.random.normal(self.repair_avg_time, self.repair_std_time)
            repair_time = max(repair_time, 0)  # 确保维修时间为正值
            
            # 更新最后一次故障时间
            self.last_breakdown_time = self.env.now  
            self.repair_time = repair_time
```

## 二、 卡车调度机制

真实的矿山调度是高度异步和分散的。该系统的仿真底层逻辑是事件驱动的。车辆在其自身状态发生切换时，主动向调度中心请求指令。

从代码实现中可以清晰地看到，卡车的调度发生在三个生命周期节点：
1. **初始状态/充电完毕**：需要去往某个装载区。
2. **装载完毕**：需要去往某个卸载区。
3. **卸载完毕**：需要返回某个装载区。

卡车在不同的地点分别调用不同的调度接口：

使得调度器（Dispatcher）能够根据卡车当下的情况，结合全局的矿山状态，做出符合当下的针对性决策。

## 三、 动作空间设计的边界

对于这个项目而言：
1. 当车辆在**充电站**（init事件）或**卸载区**（unhaul事件）时，它的下一个合理去向只能是**装载区**。如果选择去卸载区则是不合法的。
2. 当车辆在**装载区**（haul事件）时，它装满了矿石，下一步必须前往**卸载区**。

为了简化神经网络的结构，系统采用了一个统一大小的离散动作空间：
`Action Space Size = max(装载区的数量, 卸载区的数量)`


## 四、 奖励函数

如果仅仅在模拟结束时给出一个总产量作为奖励（稀疏奖励），网络极难收敛。因此项目使用了经过精心设计的**密集奖励（Dense Reward）**函数，在每一次动作后给予即时反馈。

```python
    def _get_reward_dense(self, mine: "Mine", truck: "Truck" = None, action: int = None) -> float:
        """
        最大化产量方向的密集 reward。
        
        惩罚项：
            - 车队的总排队时间（鼓励车队分散，避免扎堆）
            - 车辆的总卸载时间（鼓励前往卸载效率高、设备先进的站点）
            - 系统堵车用时（惩罚派车进入拥堵路段的行为）
            - 车辆路上的运行时间（鼓励就近原则，减少无效空驶时间）
        
        奖励项：
            - 车辆的中间产出增量，使用 log(delta + 1) 进行数值域平滑化，避免极高方差。
            - 单位时间的产量斜率奖励。
            
        :param mine: 当前矿山环境对象
        :param truck: 当前卡车对象
        :param action: 所采取的动作
        :return: 浮点数形式的即时奖励值
        """
        pass
```


- **时间即成本**：模型通过大规模对闲置时间、拥堵时间进行惩罚，避免了多辆车在同一装载点长期堵死的问题。
- **对数平滑**：产量的巨大数量级差异很容易引起梯度的剧烈爆炸，`log(delta + 1)` 变换在保留增减趋势的同时，压制了异常的大额奖励，有助于训练稳定性。

## 五、 业务问题向强化学习语言的翻译

为了让神经网络能够理解和决策，我们需要将上述复杂的工业场景抽象为标准的马尔可夫决策过程（MDP）。

1. **状态 ($S_t$)**：当一辆车需要调度时，它所“看到”的世界局势。这包括这辆车本身的属性、各个装载区/卸载区的排队长队、铲车的可用性、各路段的拥堵指数等。
2. **动作 ($A_t$)**：在此状态下，智能体做出的动作指令，即为这辆车选择的下一个目的地索引（某个装载区或卸载区）。
3. **奖励 ($R_t$)**：系统对刚刚做出的决定给出的短期/长期评价，通常体现为产量的增加，以及排队时间的扣减。
4. **终止状态 ($Done$)**：系统仿真时间到达设定的班次总时长。

## 六、 回合机制

矿山仿真是**按总时长（Sim Time）推进**的。在配置参数文件中，项目指定了一个完整的模拟回合（Episode）为 `240` 分钟环境时间，即典型的“半个班次”或“单班次”时长。

```python
# 矿山事件主循环片段
    def run(self):
        """
        矿山的仿真主进程，负责推进SimPy环境时间并判定是否终止回合。
        当系统时间 env.now 达到配置的仿真总时长后，发送 Done=True 给 RL 端。
        """
```

这种机制说明目标是进行**“班次级别优化”**，智能体要在有限的时间预算内，榨干路网和设备的潜能，跑出最大的货运量。如果是按“送完 N 车就停”，可能会导致智能体在尾声时选择躺平（只要送完即可，不再追求效率）。


## 七、 基于 CleanRL 的 PPO 模型训练

策略训练的脚本在 `openmines/test/cleanrl/ppo_single_net.py` 。

### 1) 训练入口与并行环境

脚本启动后会先计算训练的三个量：

- `batch_size = num_envs * num_steps`
- `minibatch_size = batch_size // num_minibatches`
- `num_iterations = total_timesteps // batch_size`

随后构建向量化环境：

```python
envs = gym.vector.SyncVectorEnv(
    [make_env(args.env_id, i, args.capture_video, run_name) for i in range(args.num_envs)]
)
```

这里选择 `SyncVectorEnv` 而非 `AsyncVectorEnv`，本质上是为了减少 DES（离散事件仿真）与多进程间通信带来的额外复杂度。在这个项目里，每个子环境内部本身就通过 `obs_queue` / `act_queue` 与仿真进程交互；再叠加异步向量环境，调试难度和时序不确定性会显著上升。同步并行虽然不一定“理论最快”，但在工程上更稳、更可复现。

### 2) 网络结构为共享 Actor 特征干路 + 独立 Critic 支路

`Agent` 中固定了观测维度 `obs_shape = 194`，动作维度为：

$$
|\mathcal{A}| = \max(\text{load\_sites\_num},\ \text{dump\_sites\_num})
$$

也就是前文提到的统一离散动作空间。

结构上值得注意的是：

1. Actor 走 `shared_net`（3 层 ReLU MLP）提取共享特征后输出 logits。
2. Critic 不是复用同一条干路，而是单独一条 `Tanh` MLP 输出状态价值。
3. 线性层采用正交初始化（`layer_init`），Actor 输出层使用更小的初始化方差（`std=0.01`），有助于训练初期避免策略分布过于尖锐。

```python
self.shared_net = nn.Sequential(
    layer_init(nn.Linear(self.obs_shape, hidden_size)), nn.ReLU(),
    layer_init(nn.Linear(hidden_size, hidden_size)), nn.ReLU(),
    layer_init(nn.Linear(hidden_size, hidden_size)), nn.ReLU()
)
self.actor = layer_init(nn.Linear(hidden_size, self.max_action_dim), std=0.01)

self.critic = nn.Sequential(
    layer_init(nn.Linear(self.obs_shape, hidden_size)), nn.Tanh(),
    layer_init(nn.Linear(hidden_size, hidden_size)), nn.Tanh(),
    layer_init(nn.Linear(hidden_size, 1), std=1),
)
```

### 3) 正则化机制

矿山特征中既有时间量、队列量，也有吨位量，尺度跨度大。脚本在 `Agent` 中加载 `normalization_params`，注册为 buffer：

- `obs_mean`, `obs_std`
- `reward_mean`, `reward_std`

训练时先做 `normalize_obs`，奖励可按 `r_mode` 控制：

- `reward_norm`：即时奖励标准化
- `return_norm`：对回报（returns）按 batch 在线标准化
- `none`：不做奖励标准化

这一层处理的意义是让梯度更新更多反映“决策优劣”，而不是被不同物理量纲主导。

### 4) Rollout 采样：事件驱动环境如何喂给 PPO

每次迭代会采集 `num_steps * num_envs` 条 transition。主循环里保存了 PPO 需要的全部张量：

- `obs, actions, logprobs, rewards, dones, values`
- 以及项目特有的 `time_deltas`（来自观测中的时间增量）
- 还有 `sug_actions`（建议动作，供 teacher guidance 使用）


### 5) 创新点：时间增量折扣版 GAE

标准 GAE 默认每一步时间间隔相同，但矿山是事件驱动系统，两个决策点之间的真实时间间隔并不固定。代码里因此使用了：

$$
\gamma_t = \gamma^{\Delta t_t},\quad \lambda_t = \lambda^{\Delta t_t}
$$

并据此计算 TD 误差与优势：

$$
\delta_t = r_t + \gamma_t V(s_{t+1}) - V(s_t)
$$

$$
A_t = \delta_t + \gamma_t \lambda_t A_{t+1}
$$

对应代码：

```python
gamma_dt = (args.gamma ** time_deltas[t]).to(device)
lambda_dt = (args.gae_lambda ** time_deltas[t]).to(device)

delta = rewards[t] + gamma_dt * nextvalues * nextnonterminal - values[t]
advantages[t] = lastgaelam = (
    delta + gamma_dt * lambda_dt * nextnonterminal * lastgaelam
)
```

相当于**10 分钟后才收到的收益，和 1 分钟后收到的收益，不应该按同一折现权重处理**。这比把事件离散成固定步长更贴合矿山真实节奏。

### 6) PPO 损失

更新阶段会打平 batch 并做 `update_epochs` 轮小批量优化， loss 为：

$$
\mathcal{L}=\mathcal{L}_{pg} - c_e\mathcal{H} + c_v\mathcal{L}_{v} + c_g\mathcal{L}_{guide}
$$

对应到代码即：

- `pg_loss`：裁剪后的策略目标（PPO Clip）
- `entropy_loss`：熵奖励，防止策略过快塌缩
- `v_loss`：价值损失，可选 value clipping
- `guide_loss`：教师建议动作引导项（当启用且有有效建议动作时生效）

```python
loss = (
    pg_loss
    - args.ent_coef * entropy_loss
    + args.vf_coef * v_loss
    + guide_coef * guide_loss
)
```

并且每个小批次都做梯度裁剪：

```python
nn.utils.clip_grad_norm_(agent.parameters(), args.max_grad_norm)
```

这对高方差奖励场景下的稳定训练非常重要。
